# Auth Subsystem Cleanup Plan (2026-03-02)

## Purpose
This plan critiques the current auth cleanup direction and defines a final, scoped execution plan to remove unnecessary auth behavior, fill missing functionality, and stabilize resource/profile lifecycle behavior.

This is a planning artifact only. No runtime/code changes are included here.

## Scope
In scope:
1. `/auth` UX and lifecycle behavior for API/tool credentials.
2. MCP vs `/auth` responsibility boundaries.
3. Draft generation/idempotency, duplicate profile creation, and orphan cleanup behavior.
4. OAuth ergonomics parity between CLI and TUI for `/auth` profiles.
5. Audit/repair tooling and migration-safe rollout.

Out of scope:
1. New external secret providers.
2. Full redesign of MCP transport/auth internals.
3. Breaking schema changes without migration.

## Current State Snapshot
Observed in current workspace and code paths:
1. Duplicate generated drafts exist (`*_2`) for the same semantic provider/resource.
2. Remote MCP aliases can still surface as `/auth` draft candidates despite MCP-native OAuth ownership.
3. `/auth` OAuth lifecycle is incomplete in UI/CLI compared with MCP OAuth lifecycle.
4. Opening Auth Manager mutates state via automatic draft sync.
5. Audit reports deleted bindings as orphaned noise.

## Critique Of Current Plan

### What is strong
1. Correctly identifies authority boundary issue: MCP-managed OAuth vs `/auth`-managed credentials.
2. Correctly identifies duplicate generation and idempotency concerns.
3. Correctly calls out missing OAuth UX in Auth Manager.
4. Correctly prioritizes audit/repair support over ad-hoc manual cleanup.

### What is weak or missing
1. It does not define a strict, testable boundary rule for when MCP resources are allowed into `/auth` discovery.
2. It does not specify idempotency keys and rebind criteria for draft reuse.
3. It does not separate UX-side fixes from data-model reconciliation sequencing.
4. It does not define migration behavior for already-duplicated global drafts tied to workspace-local resource IDs.
5. It does not define rollout gates and invariants that prevent further `_2` proliferation during rollout.
6. It does not specify whether auto-sync-on-open remains a product behavior or becomes explicit user intent.

### Risk in current form
1. Partial fixes could suppress future duplicates without repairing existing state, leaving users with confusing profile lists.
2. Boundary fixes could hide required resources if explicit process requirements are not preserved.
3. OAuth UX additions without metadata templates could expose non-functional login actions.

## Final Decisions
1. MCP remote aliases with MCP-native OAuth are MCP-owned by default and must not auto-create `/auth` drafts.
2. Explicit process/tool `auth.required` entries remain authoritative and can still require `/auth` resources, including MCP references.
3. Draft sync must be idempotent and reuse compatible existing drafts before creating new IDs.
4. Generated draft identity must not depend on workspace-random resource UUIDs when persisted in global `~/.loom/auth.toml`.
5. `/auth` OAuth must have complete lifecycle surface: login, status, logout, refresh in CLI and TUI.
6. Auth Manager refresh behavior must be consistent across keyboard and button paths.
7. Automatic mutation on screen-open must be removed or behind explicit user action.
8. Audit output must distinguish active-orphan vs deleted-history and avoid false-positive orphan noise.

## Invariants (Must Hold)
1. No new duplicate generated draft profiles for the same logical resource in repeated sync runs.
2. No MCP remote OAuth alias creates `/auth` draft profiles unless explicitly required by contract.
3. Every OAuth-capable `/auth` profile has a discoverable and actionable lifecycle path in both CLI and TUI.
4. Running `auth audit` on a clean workspace reports zero active orphan findings.
5. Any migration step is reversible via snapshot/rollback.

## Workstreams

## Workstream A: Boundary Hardening (MCP vs `/auth`)
Goal: Stop unnecessary `/auth` draft generation for MCP-native OAuth aliases.

Tasks:
1. Define `mcp_external_auth=true` resolution rule using MCP server type and OAuth-enabled metadata, not only command/arg heuristics.
2. Apply this rule in auth resource discovery.
3. Preserve explicit `auth.required` resource contracts as an override path.
4. Add tests for:
   - remote MCP OAuth alias skipped by default,
   - explicit requirement retained,
   - local/env-driven MCP still discoverable when applicable.

Acceptance:
1. Notion-style remote OAuth MCP alias no longer appears as autogenerated `/auth` draft.
2. Existing explicit process requirements continue to resolve.

## Workstream B: Draft Idempotency and Duplicate Suppression
Goal: Eliminate `_2` churn and reuse equivalent generated drafts.

Tasks:
1. Add compatibility matcher before draft creation:
   - same `generated_from`, or
   - same logical tuple (`resource_kind`, `resource_key`, `provider`, `mcp_server`, `mode`).
2. Rebind existing compatible draft when binding missing.
3. Only create new draft ID when no compatible reusable draft exists.
4. Add content-diff no-op guards for sync writes.
5. Add tests for repeated sync cycles and cross-workspace behavior.

Acceptance:
1. Re-running sync does not create new draft IDs for unchanged requirements.
2. Existing orphan draft is reused and rebound when possible.

## Workstream C: Identity Reconciliation (Global Profiles vs Workspace Resources)
Goal: Resolve model mismatch that causes global profile duplication tied to workspace-local UUIDs.

Tasks:
1. Stop treating workspace `resource_id` as primary global profile identity.
2. Use stable logical identity fields in generated profile metadata.
3. Add migration adapter to map old `resource_id`-only generated drafts to stable logical identity.
4. Keep backward compatibility for reading old metadata during migration window.

Acceptance:
1. Generated draft identity is stable across workspaces.
2. Migration does not break existing bindings/defaults.

## Workstream D: OAuth Lifecycle Parity for `/auth`
Goal: Make OAuth profiles usable end-to-end from Auth Manager and CLI.

Tasks:
1. Add `/auth profile status`, `/auth profile logout`, `/auth profile refresh` commands to complement existing login.
2. Add OAuth action row in Auth Manager for OAuth modes:
   - Login,
   - Status,
   - Logout,
   - Refresh.
3. Add mode-aware field guidance and validation in Auth Manager.
4. Provide provider metadata templates for known OAuth providers to make generated drafts actionable.

Acceptance:
1. User can complete OAuth flow for `/auth` profile without leaving TUI.
2. CLI and TUI provide functionally equivalent OAuth lifecycle operations.

## Workstream E: UX Behavior and Safety Cleanup
Goal: Remove surprising side effects and align control semantics.

Tasks:
1. Remove implicit draft sync on Auth Manager mount.
2. Introduce explicit `Sync` action or make both refresh paths call the same sync+refresh behavior.
3. Add visible summary of what changed during sync.
4. Ensure no silent mutation from passive view-open.

Acceptance:
1. Opening Auth Manager does not modify persisted auth state unless user explicitly requests sync/save.
2. `Ctrl+R` and Refresh button semantics are identical.

## Workstream F: Audit and Repair Tooling
Goal: Provide deterministic cleanup for existing broken/duplicate state.

Tasks:
1. Update audit classification:
   - active orphan bindings,
   - deleted historical bindings,
   - duplicate generated drafts for same logical resource,
   - stale generated metadata/resource links.
2. Add `auth repair` plan/apply flow:
   - dedupe generated drafts,
   - rebind compatible profiles,
   - archive/remove superseded drafts,
   - prune deleted-binding noise optionally.
3. Keep dry-run mode as default for repair.

Acceptance:
1. Audit output is low-noise and actionable.
2. Repair can clean current duplicate/orphan state in one guided run.

## Rollout Sequence
1. Phase 1: Workstream A + B (stop new bad state first).
2. Phase 2: Workstream C (identity reconciliation and migration adapters).
3. Phase 3: Workstream E (UX side-effect cleanup).
4. Phase 4: Workstream D (OAuth parity).
5. Phase 5: Workstream F (audit/repair hardening and cleanup tooling).

Rationale:
1. Prevent further corruption before migration and UX expansion.
2. Avoid adding OAuth UI on top of unstable draft identity behavior.

## Test Plan
Required automated coverage:
1. Discovery boundary tests for remote MCP OAuth aliases.
2. Idempotent sync tests with repeated runs and mixed legacy metadata.
3. Cross-workspace duplicate suppression tests.
4. Auth Manager interaction tests for sync/refresh semantics.
5. `/auth` OAuth lifecycle CLI tests: login/status/logout/refresh.
6. Audit/repair tests for deleted binding handling and duplicate detection.

Required verification command sets:
1. `uv run ruff check`
2. `uv run pytest tests/test_auth_config.py tests/test_tui.py tests/test_cli.py`
3. `uv run pytest` before merge

## Migration and Safety
1. Snapshot auth files before any destructive repair/migration path.
2. Keep rollback command parity with existing migrate rollback semantics.
3. Migration must be additive-first and reversible.
4. Do not delete user-created non-generated profiles automatically.

## Open Questions
1. Should generated drafts remain global (`~/.loom/auth.toml`) or move to workspace-scoped draft storage?
2. Should Auth Manager default action be `Refresh` (read-only) plus separate `Sync`, or unified `Refresh+Sync` with explicit confirmation?
3. For known providers, where should OAuth metadata templates live (package manifest vs central registry)?
4. Should deleted bindings be retained indefinitely for history, or compacted by retention policy?

## Definition of Done
1. No unnecessary MCP-native OAuth drafts are auto-generated in `/auth`.
2. Duplicate generated profile creation is eliminated under repeated sync.
3. Existing duplicate state can be repaired via supported command path.
4. `/auth` OAuth lifecycle is complete and parity exists between CLI and TUI.
5. Auth Manager behavior is explicit and non-surprising.
6. Audit output is accurate, low-noise, and actionable.
